## 分析方法

1. 逐项对比 `docs/plan/API.md` 的 API 树与方法签名 vs `src/` 的实际代码
2. 检查 CLI 定义 `vhdx-cli/src/cli.rs` vs API.md 的 CLI 命令树
3. 检查 `docs/Standard/` 合规清单是否满足
4. 分类:
   - **A 类**: 计划有 → 实现无（缺失或不同）
   - **B 类**: 实现有 → 计划无（超额实现）

## A 类：计划中有、但实现中缺失或有差异的项

> 每个 A 类项均有一个复选框，标记为 `- [ ]` 表示**待解决**，改为 `- [x]` 表示**已解决**。

- [ ] ### A1. `File::OpenOptions` / `File::CreateOptions` 非关联类型
  > **解决方式**: 修改 `lib.rs` 或更新 API.md 描述，使之一致。
- **API.md 计划** (API 树 §2):
  ```rust
  impl File {
      pub struct OpenOptions;
      pub struct CreateOptions;
  }
  impl File::OpenOptions { ... }
  impl File::CreateOptions { ... }
  ```
- **实际实现** (`src/file.rs:1782, 1843`): `OpenOptions` 和 `CreateOptions` 是模块级独立 `pub struct`，非 `File` 的关联类型。
- **影响**: 调用方式相同（`File::open(...)` / `File::create(...)` 返回构建器），但类型路径为 `vhdx_rs::OpenOptions` 而非 `vhdx_rs::File::OpenOptions`。API 文档描述不匹配。

---

- [ ] ### A2. `FileTypeIdentifier::signature()` 返回引用而非值
  > **解决方式**: 二选一——更新 API.md 计划为 `&[u8; 8]`，或修改代码返回值 `[u8; 8]`。
- **API.md 计划** (line 563):
  ```rust
  pub fn signature(&self) -> [u8; 8];     // 值语义
  ```
- **实际实现** (`src/sections/header.rs:193`):
  ```rust
  pub const fn signature(&self) -> &[u8; 8];  // 引用语义
  ```
- **影响**: 调用方需要解引用。

---

- [ ] ### A3. `FileTypeIdentifier::creator()` 返回 `String` 而非 `&[u8]` 🔴
  > **解决方式**: 建议以计划为准，修改代码返回 `&'a [u8]`；或更新计划接受 `String`。
- **API.md 计划** (line 564):
  ```rust
  pub fn creator(&self) -> &'a [u8];      // 原始字节切片
  ```
- **实际实现** (`src/sections/header.rs:201`):
  ```rust
  pub fn creator(&self) -> String;        // 已解码的 UTF-16 字符串
  ```
- **影响**: 语义差异大。计划承诺返回原始字节（调用方自行解码），实际直接返回解码后的字符串。这是 API 合约的实质性差异。

---

- [ ] ### A4. `ParentChainInfo::child()` / `parent()` 返回引用而非值
  > **解决方式**: 以实际为准（`&Path` 更优），更新 API.md 计划签名。
- **API.md 计划** (line 237-238):
  ```rust
  pub fn child(&self) -> PathBuf;          // 拥有所有权的 PathBuf
  pub fn parent(&self) -> PathBuf;
  ```
- **实际实现** (`src/file.rs:167, 173`):
  ```rust
  pub fn child(&self) -> &Path;            // 借用引用
  pub fn parent(&self) -> &Path;
  ```
- **影响**: 返回 `&Path` 比 `PathBuf` 更灵活（调用方可 `to_path_buf()` 或 `clone()`），但与计划签名不同。

- [ ] ### A7. `Sector` 缺少 `payload_ref()` 方法
  > **解决方式**: 二选一——在 `Sector` 上补 `payload_ref()` 作为 `payload()` 的别名，或更新 API.md 移除 `payload_ref`。
- **API.md 计划** (line 974-975):
  ```rust

  ```
- **实际实现** (`src/io_module.rs:246`): 只有 `payload()` 方法，无 `payload_ref()`。
- **影响**: 计划中同时存在两个方法，实际只有 `payload` 一个。

---

- [ ] ### A8. `PayloadBlock::bytes()` 是公共字段而非方法
  > **解决方式**: 以实际为准（字段更惯用），更新 API.md 将方法语法改为字段描述。
- **API.md 计划** (line 995):

  }
  ```
- **实际实现** (`src/io_module.rs:258`):

  }
  ```
- **影响**: 调用方从 `payload_block.bytes()` 变为 `payload_block.bytes`。

- [ ] ### A9. CLI `repair` 子命令未在计划中

- **影响**: CLI 命令树多了 `vhdx-tool repair <file> [--dry-run]`。

- [ ] ### A10. CLI `create --disk-type` 兼容别名

## E 类：行为级差异（不在签名中体现）

> 每个 E 类项均有一个复选框，标记为 `- [ ]` 表示**待解决**，改为 `- [x]` 表示**已解决**。

- [ ] ### E1. `SpecValidator::validate_file()` 的条件编排
  > **解决方式**: 更新 API.md，补充说明 parent 校验仅在 `has_parent=true` 时执行。

- **计划描述** (API.md line 458): "总入口：执行全部结构校验" —— 未说明条件分支。
- **实际实现** (`src/validation.rs:110-122`):
  - `validate_parent_locator()` 和 `validate_parent_chain()` 仅在 `has_parent=true` 时执行。
  - `validate_log()` 始终执行（无条件）。
- **建议**: 计划应补述条件分支逻辑：parent 校验仅适用于差分盘。

- [ ] ### E2. `CreateOptions::finish()` 的父路径校验
  > **解决方式**: 更新 API.md，补充创建时父路径存在性检查的描述。

- **计划** (API.md line 428): 描述为 "指定 parent_path 但 Parent Locator 约束不满足 -> `ParentNotFound` / `InvalidFile`"。
- **实际实现** (`src/file.rs:1911-1917`):
  ```rust
  if let Some(parent_path) = &self.parent_path
      && !parent_path.exists()
  {
      return Err(Error::ParentNotFound { path: parent_path.clone() });
  }
  ```
  在创建时立即检查父路径是否存在。计划未明确创建时的父路径校验时机。

## 综合统计

| 类别 | 数量 | 优先级 |
|------|------|--------|
| **A 类缺失/差异** | 10 项 | A3(A) / A5(A) / A7(A) / A9(A) 值得关注 |
| **B 类超额实现** | 14 组 | 多数为内部方法或合理扩展，无需修改代码 |
| **D 类类型签名差异** | 8 项 | `creator()` 返回类型差异最大 |
| **E 类行为差异** | 2 项 | 需补入计划文档 |

### 建议优先处理的差异

1. **`FileTypeIdentifier::creator()` 返回 `String` 而非 `&[u8]`** (A3) — 若调用方依赖原始字节则需修改。
2. **`Error` 缺失 `UnsupportedVersion`** (A5) — 计划文档需补入此项。
3. **`Sector` 缺失 `payload_ref()`** (A7) — 若计划为双方法设计，实际需补。
4. **CLI `repair` 子命令** (A9) — 计划 CLI 树需新增 `repair` 子命令描述。
