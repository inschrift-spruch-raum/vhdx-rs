### Backlog 条目详情

#### BL-LOG-WRITE: Log-write crash consistency（日志写入崩溃一致性）

**目标**：在 `write_dynamic` / `allocate_payload_block` 执行元数据变更前，先将变更意图写入 VHDX 日志区域（log entry），确保崩溃后可通过日志回放恢复一致性。

**当前状态分析**：
- `src/file.rs:567-605` `write_dynamic` 直接执行 `file.write_all` 写入数据并更新 BAT。
- `src/file.rs:650-692` `allocate_payload_block` 直接修改磁盘 BAT 条目 + `sync_all`，不经过日志保护。
- `src/sections/log.rs` 具备完整的日志回放能力（`Log::replay`、`replay_with_log_guid`），但无日志写入 API。
- 日志写入路径（构造 log entry + 更新 header 的 log_guid + sync）完全缺失。

**风险**：
- 崩溃（断电/进程 kill）发生在 BAT 更新后、数据写入前：BAT 指向未写入区域，读取返回垃圾数据。
- 崩溃发生在部分数据写入后：payload block 内容不完整，无检测手段。
- 对 Fixed 类型无影响（Fixed 连续写入无元数据依赖），仅 Dynamic/Differencing 受影响。

**验收草案**：
1. 新增 `Log::write_entry()` 或等效 API，能够构造日志条目（header + descriptors + data sectors）并写入日志区域。
2. `allocate_payload_block` 在修改 BAT 前，先将 {BAT 偏移, 新状态, 新偏移} 作为日志描述符写入。
3. 写入完成后清零 log_guid（通过头部更新），确认事务提交。
4. 崩溃后重新打开时，`File::open` 的日志回放能正确恢复 BAT 状态。
5. 现有测试 + 新增崩溃模拟测试全部通过。

**为何"非本轮阻断"**：
- 当前计划承诺面（T1 矩阵）未包含"日志写入"能力。回放能力已完整实现，写入路径属增强特性。
- T4 已确认 CRC-32C 校验能力完备，仅时机可评估。BL-LOG-WRITE 是校验时机之后的更高层需求。
- 实现涉及新增日志构造 API、header log_guid 生命周期管理、sync 策略，工作量大且不影响现有 API 契约。

---

#### BL-DATA-WRITE-GUID: DataWriteGuid update timing（数据写入 GUID 更新时机）

**目标**：在每次数据面写入（`write_raw`/`write_dynamic`）完成后，更新头部结构中的 `DataWriteGuid`，使该 GUID 真实反映"最后一次数据写入"时刻（MS-VHDX §2.2.2）。

**当前状态分析**：
- `src/file.rs:983-990` `init_session_header` 仅更新 `FileWriteGuid`（会话标识），保留 `current.data_write_guid()` 不变。
- `src/file.rs:1273` `create_file` 中生成初始 `data_write_guid`，但之后从不再更新。
- `src/file.rs:527-556` `write_raw` 和 `write_dynamic` 完成写入后不更新任何头部字段。
- 差分磁盘的 `parent_linkage` 使用父盘的 `DataWriteGuid` 进行链校验（`build_parent_locator_payload:1698`），若父盘 `DataWriteGuid` 不更新，差分链校验可能产生假阴性。

**风险**：
- 差分磁盘链校验：子盘记录的 `parent_linkage` 与父盘实际 `DataWriteGuid` 不一致时，`ParentChainInfo::linkage_matched` 将误报为 `false`。
- 多写入器场景：其他写入器依赖 `DataWriteGuid` 判断磁盘是否被修改，不更新将导致脏数据被误判为干净。
- 对单盘 Fixed 类型使用影响较小（无差分链依赖）。

**验收草案**：
1. `write_raw`（或 `write_dynamic`）完成数据写入后，调用头部更新函数将 `DataWriteGuid` 刷新为新 GUID。
2. 更新遵循双头交替写入策略：写入非活动头部，递增序列号。
3. 差分磁盘链校验场景下，父盘写入后 `DataWriteGuid` 变化，子盘需重新绑定或链校验正确报告 mismatch。
4. 性能影响可接受：每次 `write_raw` 不引起完整头部重写（可考虑批量更新或延迟更新策略）。
5. 现有测试全部通过。

**为何"非本轮阻断"**：
- `DataWriteGuid` 不在 T1 计划承诺面的 API 签名中。`ParentChainInfo::linkage_matched` 已实现，但计划文档未将"实时更新 DataWriteGuid"列为阻断。
- 影响范围限于差分链一致性校验，对单盘读写无功能阻断。
- 实现需要设计更新频率策略（每 write vs 每 session vs 手动），属于设计决策而非缺陷修复。

#### BL-DIFF-BITMAP: Differencing bitmap write path（差分磁盘位图写入路径）

**目标**：差分磁盘（`has_parent=true`）的 Dynamic 写入路径应正确维护扇区位图（sector bitmap），将新分配的块标记为 `PartiallyPresent` 并设置对应扇区位图的 bit，使读取路径能正确区分"子盘数据"和"回退父盘数据"。

**当前状态分析**：
- `src/file.rs:650-692` `allocate_payload_block` 对所有 Dynamic 写入统一设置 `FullyPresent`，不区分是否为差分磁盘。
- `src/file.rs:354-488` 读取路径已实现 `PartiallyPresent` + sector bitmap 的完整逻辑：按位图 bit 判定读取子盘 payload 或回退父链。
- `src/file.rs:1484-1511` `create_bat_data` 对 Dynamic 类型创建全零 BAT（所有 `NotPresent`），sector bitmap 条目也是零。
- 写入路径不分配 sector bitmap block，不设置 bit，不更新 bitmap BAT 条目。

**风险**：
- 差分磁盘新分配的块被错误标记为 `FullyPresent`，导致读取时跳过父链回退，返回错误数据（应从父盘继承的部分被子盘零数据覆盖）。
- 数据损坏：写入小块数据后，块内未写入部分不再回退父盘，返回全零而非父盘原始数据。
- 读取路径的 `PartiallyPresent` + bitmap 逻辑因写入路径从不使用而成为死路径（仅日志回放可能触发）。

**验收草案**：
1. `allocate_payload_block` 检测 `has_parent`，若为差分磁盘则分配 payload block + sector bitmap block。
2. BAT payload 条目标记为 `PartiallyPresent`，sector bitmap BAT 条目标记为 `Present`。
3. 写入时，对每个被写入的扇区设置 bitmap 对应 bit。
4. 读取路径回退行为正确：未写入扇区从父链读取，已写入扇区从子盘 payload 读取。
5. `FullyPresent` 仅在非差分磁盘（`!has_parent`）的 Dynamic 写入中使用。
6. 差分磁盘集成测试：创建父子盘 → 子盘写入部分数据 → 验证读取混合正确性。

**为何"非本轮阻断"**：
- 差分磁盘写入路径在 AGENTS.md 已知问题中已有记录（"Dynamic 块分配未完全实现"），T1 矩阵未将其列为计划承诺面阻断。
- 读取路径已完整实现 sector bitmap 逻辑，写入路径的补全是功能增强，不影响现有 API 签名。
- 实现涉及 chunk 交错规则（payload + bitmap BAT 索引计算）、bitmap block 分配与管理、与日志写入路径的协同，工作量较大。
